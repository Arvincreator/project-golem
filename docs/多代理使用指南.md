# 🤝 Project Golem 多代理使用指南（Coordinator-Worker Hard Cut）

本文件說明 Project Golem 目前正式支援的多代理協議與操作流程。  
**重點**：舊版 `multi_agent` 已硬切停用，請改用 `agent_*` 協議。

---

## 1. 核心觀念

- 多代理採 **Coordinator-Worker** 架構。
- 任務流程固定為：`research -> synthesis -> implementation -> verification`。
- 若啟用 Planning Mode，直聊會在複雜任務自動啟動多代理，不需手動輸入 `agent_session_create`。
- 啟動順序建議：
  1. 建立 session：`agent_session_create`
 2. 依角色啟動 worker：`agent_worker_spawn`（`research -> synthesis -> implementation -> verification`）
  3. 等待或查詢進度：`agent_wait` / `agent_get` / `agent_list`
  4. 需要中斷/恢復：`agent_stop` / `agent_resume`

---

## 2. 直聊自動模式（Planning Mode）

### 2.1 使用方式

- Telegram：
  - `/planning on`
  - `/planning off`
  - `/planning status`
- Web Dashboard：
  - 直聊頁右上角 `Planning Mode` toggle

### 2.2 行為規則

- 開啟後僅在「複雜任務」自動走多代理（不是每句都走）。
- 使用者不需要手動輸入 `agent_session_create`。
- 中間 worker 思考不會出現在直聊 / TG 視窗，只顯示主 Golem 最終回報。
- 詳細過程可在 Dashboard Agents 事件流查看與審計。

## 3. 快速開始（Action Protocol）

### 建立 session

```json
{
  "action": "agent_session_create",
  "input": {
    "objective": "完成 API 錯誤碼一致化與回歸測試",
    "strategy": "先研究現況，再實作，最後驗證"
  }
}
```

### 啟動 research worker

```json
{
  "action": "agent_worker_spawn",
  "input": {
    "sessionId": "agent_session_000001",
    "role": "research",
    "prompt": "盤點現有錯誤碼與 route 回傳格式"
  }
}
```

### 等待進度

```json
{
  "action": "agent_wait",
  "sessionId": "agent_session_000001",
  "timeoutMs": 30000
}
```

### 啟動 synthesis worker（strict mode 建議必做）

```json
{
  "action": "agent_worker_spawn",
  "input": {
    "sessionId": "agent_session_000001",
    "role": "synthesis",
    "prompt": "把 research 結果彙整成可執行實作計畫"
  }
}
```

### 恢復未完成 session

```json
{
  "action": "agent_resume"
}
```

## 4. 完整 action 一覽

- `agent_session_create`
  - 用途：建立多代理 session
  - 主要欄位：`objective`, `strategy`, `metadata`, `maxWorkers`
- `agent_worker_spawn`
  - 用途：為 session 建立 worker
  - 主要欄位：`sessionId`, `role`, `prompt`, `runInBackground`, `model`
- `agent_message`
  - 用途：向 session/worker 送訊息
  - 主要欄位：`sessionId` 或 `workerId`, `message`
- `agent_wait`
  - 用途：等待 session 狀態更新
  - 主要欄位：`sessionId`, `timeoutMs`
- `agent_stop`
  - 用途：停止 session 或 worker
  - 主要欄位：`sessionId` 或 `workerId`, `reason`
- `agent_list`
  - 用途：列出 sessions/workers
  - 主要欄位：`status`, `statuses`, `sessionId`, `limit`
- `agent_get`
  - 用途：查詢單一 session/worker
  - 主要欄位：`sessionId` 或 `workerId`
- `agent_resume`
  - 用途：恢復可續跑的 session
  - 主要欄位：`sessionId`（可選）
- `agent_focus`
  - 用途：聚焦單一 session 的 orchestration 下一步建議
  - 主要欄位：`sessionId`

## 5. REST API 用法

以下以 `golem_A` 為例。

- `GET /api/agents/sessions?golemId=golem_A`

## 7. 跨對話恢復行為

- `/new`、worker recycle、process restart 後，系統會送出 `agent_recovery`。
- 啟動與恢復時可透過 `GET /api/agents/resume-brief` 取得：
  - 已恢復 session 數量
  - 正在執行的 worker 數量
  - 建議下一個焦點 session
- 建議先 `agent_resume` 再開新 session，避免任務分裂。

## 8. 人話回覆模式（預設）

系統預設以自然語言回報進度，不主動展示內部 `taskId/sessionId` 或 JSON 細節。  
若你要除錯，可明確要求：

- 「請顯示詳細 task/session id」
- 「請用 detail 模式輸出」

## 9. 常見錯誤與排查

### `AGENT_PROTOCOL_UNSUPPORTED`

- 原因：仍在使用舊 `multi_agent`
- 解法：改用 `agent_session_create` 等 `agent_*` action

### 任務顯示有執行但沒真正跑

- 先檢查是否被安全層判定為 `WARNING/DANGER`
- 唯讀網路查詢可用：
  - `GOLEM_TRUST_READONLY_NETWORK_COMMANDS=true`（預設）

### `AGENT_VERSION_CONFLICT` (409)

- 原因：更新時版本過舊
- 解法：先 `agent_get` 取得最新版本再 `PATCH`

### `AGENT_MAX_WORKERS_EXCEEDED` (422)

- 原因：超過 `GOLEM_AGENT_MAX_WORKERS`
- 解法：停止/完成部分 worker 或提高上限

### `AGENT_INVALID_TRANSITION`（synthesis gate）

- 原因：strict mode 下，尚未完成 synthesis 就嘗試啟動 implementation
- 解法：先建立 `role=synthesis` worker，待其 terminal 且至少一個 completed，再啟動 implementation
